# 集群调度模拟器

离散事件驱动的集群任务调度模拟器。模拟由 M 台机器构成的集群，支持两种任务类型、机器故障/维修、多种调度策略及基于拥塞度的自适应策略切换。

---

## 1. 环境要求

|------|---------|------|

## 3. 启动方式

所有命令在项目根目录下执行，入口文件为 `src/main.py`。

### 3.1 默认运行（自适应策略，10台机器，7天）

```bash
python src/main.py
```

### 3.2 自定义参数

```bash
python src/main.py --machines 20 --days 14 --normal-tasks 20 --priority-tasks 8
```

### 3.3 指定调度策略

```bash
python src/main.py --policy fifo
python src/main.py --policy priority_first
python src/main.py --policy sjf
python src/main.py --policy adaptive    # 默认
```

### 3.4 对比所有策略

```bash
python src/main.py --compare
```

使用相同随机种子依次运行 4 种策略，输出对比表格和各自的图表。

### 3.5 完整参数列表

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `--machines` | int | 10 | 集群机器数量 |
| `--days` | int | 7 | 模拟天数 |
| `--normal-tasks` | int | 15 | 每天 8:00 发布的普通任务数 |
| `--priority-tasks` | float | 5 | 每天平均优先任务数（泊松分布） |
| `--failure-rate` | float | 0.03 | 每台机器每天故障概率 |
| `--policy` | str | adaptive | 调度策略：fifo / priority_first / sjf / adaptive |
| `--compare` | flag | - | 启用后对比所有策略 |
| `--seed` | int | 42 | 随机种子，相同种子保证结果可复现 |

### 3.6 运行测试

```bash
pytest tests/ -v
```

## 4. 使用流程

```
配置参数 → 预生成事件 → 逐分钟模拟 → 输出统计 → 生成图表
```

1. **配置参数**：通过命令行参数或直接构造 `SimConfig` 对象设定机器数、模拟天数、任务参数、故障率、策略。
2. **预生成事件**：模拟器初始化时一次性生成全部任务到达事件和机器故障事件，确保相同种子完全可复现。
3. **逐分钟步进**：每一分钟依次执行 6 个阶段——维修检查 → 任务完成 → 故障检查 → 任务到达 → 调度分配 → 指标收集。
4. **输出统计**：模拟结束后打印统计摘要（完成数、等待时间、周转时间、利用率等）。
5. **生成图表**：输出 PNG 图片，包含队列长度变化、机器状态分布、等待时间直方图、统计摘要。

## 6. 设计决策说明

### 6.1 模拟方式：离散时间步进（按分钟）

选择按分钟步进而非事件驱动，原因：

- 机器故障是按概率每分钟独立触发的，天然适合时间步进。
- 实现简单直观，每步依次执行 6 个固定阶段，便于调试和理解。
- 对于本项目规模（万级分钟数 × 十级机器数），性能完全满足需求。

### 6.2 任务模型：两种类型，统一数据结构

普通任务和优先任务共用同一个 `Task` 类，通过 `TaskType` 枚举区分。这样调度器排序时只需检查一个字段，避免了继承层次的复杂度。

关键字段设计：
- `remaining_time`：支持任务被故障中断后恢复执行，只需跑完剩余时间。
- `total_wait_time` + `_last_queue_time`：累计追踪等待时间，解决被抢占任务多次入队导致等待时间被覆盖的问题。

### 6.3 机器状态机：三状态

```
IDLE ──assign_task──→ BUSY ──complete_task──→ IDLE
  │                    │
  └──── fail() ────────┘──→ REPAIRING ──24h──→ IDLE
```

- 故障直接进入 REPAIRING（维修 24 小时），不设独立的 FAILED 中间态——业务上故障即送修，无需区分。
- 维修中的机器不会再次触发故障事件（模拟引擎中跳过）。

### 6.4 调度策略：四种 + 自适应

| 策略 | 排序规则 | 适用场景 |
|------|---------|---------|
| FIFO | 严格按到达时间 | 轻负载，公平性优先 |
| PRIORITY_FIRST | 优先任务在前，同类型按到达时间 | 需要保障优先任务 SLA |
| SJF | 优先任务在前，同类型按剩余时间 | 重负载，最大化吞吐量 |
| ADAPTIVE | 根据拥塞度自动切换上述三种 | 负载波动场景（默认推荐） |

自适应切换逻辑基于**拥塞度 = 等待任务数 / 可用机器数**：

| 拥塞度 | 选择策略 | 理由 |
|--------|---------|------|
| < 0.5 | FIFO | 轻负载，无需复杂排序 |
| 0.5 ~ 2.0 | PRIORITY_FIRST | 中等负载，保障优先任务不被饿死 |
| > 2.0 | SJF | 重负载，短任务优先最大化单位时间完成数 |

阈值 0.5 和 2.0 定义在 `scheduler.py` 中的 `CONGESTION_LOW` / `CONGESTION_HIGH` 常量，可按实际需求调整。

### 6.5 故障与抢占

- 机器故障是预生成的：初始化时按 `故障概率/1440` 的每分钟概率为每台机器采样，生成故障时间表。
- 故障发生时，运行中的任务被中断，`remaining_time` 减去已执行时间后放回调度队列。
- 这保证了任务不会因故障而丢失，只是延迟完成。

### 6.6 优先任务到达：泊松分布

每天的优先任务数量服从泊松分布（通过 Knuth 算法实现），到达时间在一天 1440 分钟内均匀随机分布。这比固定数量更贴近真实场景中随机事件的统计特性。

### 6.7 指标与可视化

`MetricsCollector` 每分钟记录一条 `StepMetrics`（队列长度、机器状态分布），模拟结束后可按需聚合。图表以每小时采样避免数据点过密。

统计摘要包含：任务到达/完成数、平均等待/周转时间、故障/中断次数、集群利用率、未完成任务数。

### 6.8 测试策略

89 个测试用例分三类：

- **正向流程**（46 例）：验证正常业务逻辑——任务生命周期、调度排序、策略切换、故障恢复、确定性复现。
- **异常输入**（20 例）：0 任务/0 机器/负数参数、空队列调度、极端高负载。
- **边界测试**（23 例）：拥塞阈值精确命中、维修时间差 1 分钟、机器状态总和守恒、单机单任务最小配置。
